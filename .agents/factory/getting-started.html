From chatbots to a software factory

A guide to agentic software engineering for research software engineers — what an agent really is, how a harness turns context into action, and how a lightweight, auditable methodology (the .agents/factory) makes that power repeatable, reviewable, and safe.

~20 min read Executive orientation ~5 min Audience: RSEs & leadership Self-contained · works offline

Executive orientation

Why this, why now #

For a decade our tools have answered questions. In the last two years they began to do work. That is the whole story — and it changes how research software gets built.

A chatbot produces text. An agent is given a goal, a set of tools, and the latitude to use them in a loop: it reads your code, runs your tests, edits files, drives the CLI, inspects the result, and tries again — narrating and course-correcting as it goes. The thing that made this leap possible is not a bigger model alone; it is the harness — the software (Claude Code, Warp, Cursor, and others) that assembles a critical mass of context and grants the model the ability to act. Context plus action, in a loop, is where the wellspring of productivity comes from.

That power needs a channel. Left unstructured, an agent will happily build the wrong thing quickly, or the right thing in a way you cannot review. The discipline that channels it is method — and it turns out the good parts of agile practice (especially 37signals' Shape Up) map cleanly onto agent work. HyperShell packages that method as a small, transparent toolkit in .agents/: five lifecycle skills — plus two operational siblings, hs-harness and hs-release — and a set of reference documents that carry a change from a shaped goal, through a researched plan, through execution, through an adversarial review, to a pull request — with a durable, committed record at every step.

What changes for RSE teams #

The level of abstraction rises. People spend less time typing boilerplate and more time doing the things only people can do well: shaping what is worth building, judging whether what came back is correct, and owning the result. The agent executes; the human sets direction and holds the bar.

What the factory is

  • A repeatable lifecycle: shape → plan → build → review → ship.
  • Plain Markdown + git + skills — no new platform, no lock-in.
  • An auditable trail: intent, design, and code, committed together.
  • Human judgement at every gate that matters.

What it is not

  • Not "the AI writes everything and we hope." Every step is reviewable.
  • Not a replacement for engineers — it raises what they work on.
  • Not a heavyweight process forced onto one-line fixes (it scales down).
  • Not a frozen spec: the plan is a living artifact, the goal is the contract.

For institutional leadership, at a glance #

The four questions leaders ask most, answered in one line each — with a fuller treatment in Notes for institutional leadership.

Governance & guardrails

Human sign-off gates, blast-radius controls, and an independent adversarial review step are engineered into the lifecycle — not bolted on.

Cost & effort economics

Ceremony scales to the size of the change (an appetite), so a bug fix does not pay for a feature's process. You spend rigor where it earns its keep.

Reproducibility & auditability

Every feature leaves a committed record — goal, design, and code, traceable requirement → commit. A research-grade paper trail for how software came to be.

Workforce

This augments RSEs. Humans shape, judge, and own; agents execute. The scarce skill becomes good judgement about software, which your people already have.

Practitioner guide · 1

Agents, not chatbots #

A chatbot is a function from prompt to text. You ask; it answers; the transaction ends. Useful, but it does nothing — you still copy the snippet, run it, read the error, and come back.

An agent closes that loop itself. Given a goal and a set of tools, it runs a cycle:

01
Perceive
read files, run a command, inspect state
→
02
Reason
decide the next concrete action
→
03
Act
edit code, run tests, call a tool
→
04
Observe
read the result, then loop

The difference is not eloquence; it is agency. Because it observes the outcome of each action, an agent can recover from a failing test, chase down a stack trace, or notice that the thing it just built doesn't actually run. This is why "it wrote code" (2022) and "it shipped a reviewed change" (2026) are different eras.

Chatbot

# one shot, no hands
you   ❯ how do I add a flag?
bot   ❯ here's some code…
you   ❯ # you run it, it breaks,
        # you paste the error back…

Agent

agent ❯ read src/…/submit.py
agent ❯ edit: add --dry-run
agent ❯ run: uv run pytest -k submit
agent ❯ # red → fix → green → commit
The catch

Agency cuts both ways. An agent can build the wrong thing, or the right thing unreviewably, just as fast as the right thing well. Everything after this section is about channelling that agency — giving it the right context, the right method, and the right gates.

Practitioner guide · 2

The harness: where context becomes action #

The model is the engine; the harness is the vehicle. A coding harness (Claude Code, Warp, Cursor, Aider, and peers) does two jobs at once, and their combination is the productivity unlock:

Context in

A critical mass of context

The harness feeds the model far more than your prompt:

  • The repository — read on demand, searched, navigated.
  • Project memory — an AGENTS.md that states the conventions and invariants of this codebase.
  • Tools — the shell, tests, git, the web, databases.
  • Skills — packaged procedures the agent can invoke by name.
Action out

The ability to act

And it lets the model do things, under permission control:

  • Edit and create files across the project.
  • Run commands — tests, builds, the CLI itself.
  • Use git; open pull requests; call external services.
  • Spawn sub-agents to research in parallel.

Put a loop around those two and you have the whole machine: gather context → decide → act → observe → repeat. The art — "context engineering" — is getting the right context in front of the model at the right moment, because a model's attention is finite and its session memory is not durable across restarts.

The load-bearing idea

Because a fresh session starts blank, durable state must live in files and git, not in the chat. That single fact is why the factory writes everything down — the goal, the plan, the roadmap, the review — and reads it back fresh each time. It is what lets a build resume next week on another machine, and lets a reviewer start with no memory of how the code was written.

Skills: procedures the agent can run #

A skill is a Markdown file (SKILL.md) with a little front-matter and a body of instructions. When you type its name as a slash command — say /hs-build — the harness loads those instructions, injects live facts (the current branch, the git status), and the agent follows the procedure. Skills are how you capture "the way we do X here" once, so nobody re-explains it every time. The lifecycle skills in this guide are exactly that: the HyperShell way to take a change from idea to merge.

Practitioner guide · 3

Method: Shape Up meets agents #

Give a capable agent no method and you get motion without direction. The good news: much of what software teams already learned about directing work applies — you just keep the thinking tools and drop the calendar rituals.

The sharpest fit is Shape Up (Ryan Singer / 37signals). Its core ideas translate almost directly to a solo-engineer-plus-agent pipeline:

Shape Up ideaIn the factory
Appetite — fix the time, vary the scope ("start with a number, end with a design"). An appetite: small|big field sizes how much research and how many phases a change gets. The plan fits the appetite, not the reverse.
Shaping — rough, solved, and bounded, at the right altitude. The GOAL.md is shaped: concrete enough to de-risk, abstract enough to leave design freedom.
No-gos — say what you will not do. An explicit non-goals list keeps scope from creeping.
Rabbit holes — name the scary unknowns and settle them before committing. Each research brief investigates one unknown that could blow the appetite.
Hill — uphill (figuring out) vs downhill (just doing). Each phase carries a hill signal; a phase stuck "uphill" across builds is a raised hand.
Circuit breaker — don't extend indefinitely; stop and re-shape. Build iterations and review bounce-backs are capped; on trip, stop and re-plan.

What we deliberately discard: the betting table, six-week cycles, cool-down, and separate shaper/builder/QA roles — those exist to synchronize a human team and dissolve for a serial solo-plus-agent flow.

One place Shape Up must not be followed

Shape Up assumes scope is always cuttable and most bugs can wait. That is false for a distributed execution engine: a silent double-run or a resurrected task is not a deferrable bug. The factory marks correctness- and security-critical work as hammerable: false — the reviewer may never trim it to "fit the appetite."

The broader movement this belongs to — committing a specification alongside the code and letting the spec drive the work — is often called spec-driven development. Tools like GitHub Spec Kit and AWS Kiro converge on the same what/why → how → ordered-work spine the factory uses. HyperShell's version is deliberately lightweight: Markdown, git, and skills, anchored on the project's own AGENTS.md.

The .agents/factory lifecycle #

One change flows through five lifecycle skills, on its own git branch, leaving a committed trail under spec/<slug>/. The whole pipeline:

/hs-feature
Shape
GOAL.md — what & why
→
/hs-plan
Design
research → PLAN → TECH
→
/hs-build
Execute
one phase at a time ⟲
→
/hs-review
Judge
adversarial QA ⟲
→
/hs-publish
Ship
PR → develop

The ⟲ steps loop: review sends confirmed problems back to build until the change is clean, then publish opens the pull request. A silent +1 rides alongside — the factory improves itself, below.

The artifact spine #

Each stage reads the previous stage's committed files and writes its own. This is the durable substrate from §2 in practice:

FileWritten byWhat it is
GOAL.md/hs-feature The locked contract: problem, appetite, non-goals, and testable acceptance criteria with stable IDs (R1, R2…).
PLAN.md/hs-plan The authoritative design: how it will be built, mapped back to each requirement ID.
research/*.md/hs-plan Briefs from parallel read-only sub-agents — one per rabbit hole (big appetite only).
TECH.md/hs-plan The phased roadmap — a small state machine (see below) the build resumes from.
REVIEW.md/hs-review Severity-ranked findings + a requirement→evidence matrix.
META.mdthe lifecycle skills Optional harness-feedback log: what worked well, and where the skillset itself caused friction. Feeds the self-improvement loop (see The factory improves itself, below), not the change.

TECH.md is a finite-state machine #

The roadmap's YAML front-matter is the resume point. It lists phases (vertical slices of work), each with the requirements it satisfies, a verify command, and honesty signals. The build skill reads it fresh every time, does the next actionable phase, and writes the state back — so a build is fully resumable and git history becomes the checkpoint log.

spec/<slug>/TECH.md — front-matter (excerpt)
appetite: big              # small|big → caps phases & iterations
status: in_progress
current_phase: P2
phases:
  - id: P1
    name: "Source table + task fingerprints"
    satisfies: [R1, R3]        # traces back to GOAL requirement IDs
    hammerable: false          # correctness — review may not cut this
    hill: downhill            # uphill=figuring out · downhill=executing
    status: done
    verify: "uv run pytest -m unit -k source"

A pair of small helper scripts (next_phase.py / set_phase.py) read and update this block, so the agent never hand-edits fragile YAML.

Build in the open #

All of this — the skills, the reference docs, the per-feature spec/ folders — is committed to the repository. The methodology is not a hidden prompt; it is version-controlled, reviewable, and improvable like any other part of the codebase. The full rationale lives in methodology.md.

The factory improves itself #

The five lifecycle skills don't just run the process — they watch it for friction. Each ends with a silent-by-default meta-note step: when a concrete problem in the skillset itself — a misleading instruction, a wrong command, a gap in guidance — costs something during a job, the skill records a terse finding in that job's spec/<slug>/META.md, gated by a single test: "was this the instructions' fault, not mine or the task's?" A merely hard task is not a finding. These F# findings (plus an optional "what worked well" note) run orthogonal to the GOAL → REVIEW content spine, and are deliberately hidden from the blind reviewer (they can leak author intent).

The loop is asymmetric by design — cheap to observe, deliberate to act — so it can never become a token-sink or quietly loosen a guardrail:

observe
meta-note
skills log friction to META.md (F#)
→
surface
/hs-publish
open findings → a PR "🔧 Harness feedback" block
→
act
/hs-harness
human-gated fix to .agents/

One skill, /hs-harness, stands outside the lifecycle and is the only one that edits the factory itself. Pointed at a finished job's META.md, it shapes the fix with you, previews a diff per finding, and — only with human approval — applies each to .agents/ as an atomic [harness] commit, recording every decision in a cross-job ledger, harness-log.md. It is bound by rules it cannot break: the observer is never the fixer; it never auto-weakens a non-negotiable gate (the tests, the CLI-drive verify, or an invariants.md rule — a hammerable:false phase); fixes must generalize (no overfitting to one job); and it writes no findings of its own (no meta-on-meta). Every change lands through human review and a PR to develop — it never touches product code, and it runs on any harness (portability.md).

Practitioner guide · 5

Hands-on: the skills #

Each is a slash command you invoke deliberately (none fire on their own). They guard their own preconditions — a wrong branch or a dirty tree stops them cold — so the process is hard to run out of order.

/hs-feature — shape the goal

From a clean develop, creates a feature/<slug> or fix/<slug> branch and refines your idea (or a hand-written GOAL) into a shaped GOAL.md: appetite, non-goals, and testable criteria. Shaping only — no research, no code. Asks you when something is ambiguous rather than guessing.

reads your prompt / an untracked GOAL.md · writes spec/<slug>/GOAL.md · skill hs-feature/SKILL.md
/hs-plan — research & design

Checks the idea against the project's invariants, fans out read-only research sub-agents (scaled to appetite; codebase-first), then writes the PLAN.md design and the TECH.md phased roadmap. A small fix can skip the research entirely.

reads GOAL.md, AGENTS.md, the codebase · writes research/, PLAN.md, TECH.md · skill hs-plan/SKILL.md
/hs-build — execute the roadmap

The resume-and-implement driver. Reads the FSM, does the next phase, runs that phase's verify command (it proves work by driving the CLI, not by ticking a box), and makes one atomic commit of code + state. It may revise the plan as reality demands, but a contradiction with the locked GOAL stops it for you.

reads TECH.md, PLAN.md, the code · writes source, docs, TECH.md state · skill hs-build/SKILL.md
/hs-review — adversarial QA

Independent quality assurance. It delegates to a fresh reviewer with no memory of how the code was written, who sees only the goal, the diff, and the project's invariant checklist — not the author's own plan. It must reproduce each problem with a real command before reporting it. Confirmed issues loop back to build; changes to the risky core always require a human sign-off.

reads GOAL.md + diff + invariants (blind to PLAN/TECH) · writes REVIEW.md, findings · skill hs-review/SKILL.md
/hs-publish — ship it

Opens a pull request to develop with a rich, artifact-linked description (or does a local squash-merge). This is the one irreversible step, so it always confirms with you first and never targets the release branch.

reads all artifacts + git log · writes a PR (or a local merge) · skill hs-publish/SKILL.md
/hs-harness — the self-improvement skill (operational · not part of the lifecycle)

Stands apart from the five above: it changes the factory, not any feature. Pointed at a finished job's META.md, it shapes the fix with you, previews a diff per finding, and — only with your approval — applies each recorded process fix to .agents/ as an atomic [harness] commit, logging every decision. It never fires on its own (disable-model-invocation), never auto-weakens a non-negotiable gate, and writes no findings itself. See The factory improves itself for the loop it closes.

reads spec/<slug>/META.md + harness-log.md · writes .agents/ fixes, harness-log.md · skill hs-harness/SKILL.md
/hs-release — cut a version (operational · not part of the lifecycle)

The other operational sibling: it changes the release, not any feature. In one of three modes — patch (hotfix off master), pre-release (alpha/beta/rc off develop), or full-release (promote develop → master) — it bumps the single version source, rebuilds the man pages, runs a CI-mirror gate (tests, build, twine check --strict, sdist hygiene), signs an annotated tag, and — only after you approve, before any irreversible push — publishes to PyPI and ghcr, then verifies the result. Every branch op is rehearsed in an isolated git-worktree dry-run first. Fills the gap /hs-publish leaves.

reads pyproject.toml, git history, CI workflows · writes a version bump, man pages, a signed tag, a GitHub release · skill hs-release/SKILL.md

A day in the life #

a small feature, start to finish
/hs-feature add a --dry-run flag to hs submit that prints the plan
# → branch feature/submit-dry-run, spec/submit-dry-run/GOAL.md — you review it

/hs-plan
# → research/, PLAN.md, TECH.md — you approve the design

/hs-build            # phase P1 … commit
/hs-build            # phase P2 … commit  (repeat until done)

/hs-review           # fresh, blind QA → REVIEW.md (approved / changes-requested)

/hs-publish          # opens the PR to develop, links every artifact
Scales down

A one-line fix does not need all of this. Set appetite: small (the default for fix/ branches) and /hs-plan collapses to a lean plan with no research fan-out. A truly trivial change can skip the lifecycle altogether. Rigor is a dial, not a toll.

For institutional leadership

Notes for institutional leadership #

The at-a-glance panel above, expanded. These are the concerns that decide whether a department can adopt agentic engineering responsibly.

Governance & guardrails #

Autonomy is bounded by design, not by hope:

  • Human gates at the decisions that matter. A human approves the goal before the expensive planning, the design before code is written, and the review before anything ships.
  • Independent review. The QA step runs in a fresh context, blind to the author's reasoning, and must reproduce a defect before flagging it — a direct countermeasure to a model rubber-stamping its own work.
  • Blast-radius controls. Any confirmed problem touching the fault-tolerant core (the task database, the scheduler, the network queue, the concurrency machinery) forces a mandatory human sign-off, regardless of anything else.
  • Encoded invariants. The non-negotiable correctness and security rules of the codebase live in invariants.md and are checked at both planning and review; violating one is an automatic top-severity finding.

Cost & effort economics #

Agent work consumes tokens (and therefore money and time), most of all during broad research fan-out. The factory treats spend as a dial:

  • Appetite scaling. Small changes skip research and collapse the paperwork; only genuinely large features pay for parallel investigation. You do not fund a feature's ceremony for a typo.
  • Parallel where it pays, serial where it must. Research fans out (large, well-understood gains); building stays single-threaded (parallel builders make conflicting decisions). The expensive move is used only where it is safe and worthwhile.
  • Circuit breakers. A stuck phase or a review that keeps bouncing trips a stop, rather than burning budget in a loop — a spend ceiling with a human off-ramp.

Reproducibility & auditability #

This is where a research context benefits most. Every feature leaves a committed, dated record:

  • A traceable chain from requirement ID → design element → implementing commit → review evidence. You can answer "why does this code exist, and who decided?" from the repository alone.
  • An ADR-style trail. The spec/<slug>/ folder is retained on merge as a point-in-time record of intent and design — not a living doc to maintain forever, but a durable account of how the software came to be.
  • The code remains ground truth. Specs record intent; the code and AGENTS.md are authoritative. This avoids the classic trap of stale documentation pretending to be the source of truth.

Workforce #

The honest framing: this raises the level of abstraction at which your engineers work. The routine typing is increasingly automated; the durable, human contributions become more valuable, not less:

  • Shaping — deciding what is worth building and bounding it well.
  • Judgement — knowing whether a result is correct, safe, and maintainable.
  • Ownership — standing behind what ships.

These are exactly the skills experienced RSEs already have. The change is that they spend more of their day exercising them and less on boilerplate. It is a force multiplier for a small research software team, not a substitute for one.
